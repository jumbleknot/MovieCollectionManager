# Volume & Network Rename — Migration Runbook (proposal)

Executes the [resource-naming-convention.md](resource-naming-convention.md) rename on an existing local stack **without losing data**. One-time maintenance operation; do it with the stack idle.

> **Names: see the convention doc (v3) as source of truth.** Some example commands below still show the earlier `bff-store-…`/`bff-cache-…` names — under v3 the BFF volumes are `mcm-bff-store-mongo-data` and `mcm-bff-cache-redis-data` (only the BFF carries the `mcm-` qualifier). The migration *procedure* (backup → copy → cutover) is unchanged; the authoritative per-volume command list will be regenerated by `/speckit-plan` → `/speckit-tasks`.

## ⚠️ The core risk

A Docker volume's **name is its identity**. Changing `name:` in compose does **not** rename — Docker creates a *new empty* volume and orphans the old data. So every **stateful** volume must be **copied**, not renamed. The three that hold real data:

| Volume | Holds | If you skip the copy |
|---|---|---|
| `keycloak-store-postgres-data` ← `localdev-auth_keycloak-db-data` | the **grumpyrobot realm**, users, client secrets | login broken; realm gone |
| `mc-service-store-mongo-data` ← `mc-service_mc-db-data` | movies + collections | empty catalogue |
| `bff-store-mongo-data` ← `mcm-bff-db-data` | per-user encrypted agent configs | users must re-enter creds |

`bff-cache-redis-data`, `movie-assistant-store-postgres-data`, `…-audit-…`, `…-models-ollama-…` carry only disposable/re-derivable data → recreate empty, no copy.

## Phase 0 — Safety backup (do this even though copy is non-destructive)

```bash
mkdir -p E:/tmp/volbackup
# tar each stateful volume to a host file (read-only mount)
docker run --rm -v localdev-auth_keycloak-db-data:/v:ro -v E:/tmp/volbackup:/b alpine tar czf /b/keycloak-db.tgz -C /v .
docker run --rm -v mc-service_mc-db-data:/v:ro       -v E:/tmp/volbackup:/b alpine tar czf /b/mc-db.tgz       -C /v .
docker run --rm -v mcm-bff-db-data:/v:ro             -v E:/tmp/volbackup:/b alpine tar czf /b/bff-db.tgz      -C /v .
```
(You already have a Keycloak realm JSON export at `E:/tmp/jumbleknot-realm-backup.json` as a second safety net.)

## Phase 1 — Bring the whole stack down (release the volumes)

```bash
cd e:/Programming/VSCode/MovieCollectionManager
node scripts/agent-stack.mjs --down        # agent gateway + MCP containers (docker run, not compose)
docker compose --profile app --profile keycloak --profile bff-dev --profile audit down
docker ps    # confirm nothing from this project is still running
```

## Phase 2 — Create the new volumes + copy the stateful three

```bash
# new (empty) volumes
docker volume create keycloak-store-postgres-data
docker volume create mc-service-store-mongo-data
docker volume create bff-store-mongo-data
docker volume create bff-cache-redis-data
docker volume create movie-assistant-store-postgres-data
# (optional, only if you use them) :
# docker volume create movie-assistant-models-ollama-data
# docker volume create movie-assistant-audit-opensearch-data

# copy data for the stateful three (cp -a preserves perms/owners/timestamps)
docker run --rm -v localdev-auth_keycloak-db-data:/from:ro -v keycloak-store-postgres-data:/to   alpine sh -c 'cp -a /from/. /to/'
docker run --rm -v mc-service_mc-db-data:/from:ro          -v mc-service-store-mongo-data:/to    alpine sh -c 'cp -a /from/. /to/'
docker run --rm -v mcm-bff-db-data:/from:ro                -v bff-store-mongo-data:/to           alpine sh -c 'cp -a /from/. /to/'

# sanity: sizes should match
docker run --rm -v localdev-auth_keycloak-db-data:/a:ro -v keycloak-store-postgres-data:/b:ro alpine sh -c 'du -sh /a /b'
```
(`bff-cache-redis-data` and `movie-assistant-store-postgres-data` are intentionally left empty — sessions and checkpoints regenerate.)

## Phase 5 — Bring up against the new volumes + verify

```bash
docker compose --profile keycloak --profile app --profile bff-dev up -d
node scripts/agent-stack.mjs            # agent stack (uses the new network)

# --- VERIFY (must all pass) ---
# Keycloak realm intact:
curl -s http://localhost:8099/realms/grumpyrobot/.well-known/openid-configuration | grep -o '"issuer":"[^"]*"'
# Movies/collections preserved:
docker exec mc-db mongosh --quiet --eval "const d=db.getSiblingDB('mc_db'); print('collections='+d.movie_collections.countDocuments({})+' movies='+d.movies.countDocuments({}))"
# BFF agent-config store preserved:
docker exec mcm-mcm-bff-db-1 mongosh --quiet --eval "print(db.getSiblingDB('bff_db').getCollectionNames())"
# Redis (empty, just reachable):
docker exec mcm-mcm-redis-1 redis-cli ping
# Volumes actually in use are the new ones:
docker inspect mc-db --format '{{range .Mounts}}{{.Name}} {{end}}'
```
Compare the Mongo counts to what you recorded before (or just confirm they're non-zero and match expectations). Then run a quick smoke: `E2E_BFF_TARGET=dev-container pnpm nx e2e mcm-app -- auth.spec.ts`.

## Phase 6 — Decommission the old volumes (only after verification passes)

```bash
docker volume rm localdev-auth_keycloak-db-data mc-service_mc-db-data mcm-bff-db-data mcm-redis-data agent-db-data
# ollama-models / opensearch-data too if they existed and you migrated them
docker network rm agent-mcp
docker volume ls   # confirm only the new names remain
```
Keep `E:/tmp/volbackup/*.tgz` for a few days, then delete.

## Rollback

The old volumes are untouched until Phase 6. To revert: `docker compose down` + `node scripts/agent-stack.mjs --down`, `git restore` the compose/script edits (or check out the pre-migration commit), `docker compose up -d`. Data is exactly as it was. If you already removed the old volumes, restore from `E:/tmp/volbackup/*.tgz`:
```bash
docker run --rm -v localdev-auth_keycloak-db-data:/v -v E:/tmp/volbackup:/b alpine sh -c 'tar xzf /b/keycloak-db.tgz -C /v'
```

## Suggested sequencing

This is a repo-wide standard + a stateful migration — a good fit for its own SDD feature (e.g. `019-resource-naming`): land the convention doc + the compose/script/CI/doc edits in a PR (CI just needs the volume/network-create loops updated), then run Phases 0–6 locally as the cutover. The PR itself changes no running state; only the local cutover does.
